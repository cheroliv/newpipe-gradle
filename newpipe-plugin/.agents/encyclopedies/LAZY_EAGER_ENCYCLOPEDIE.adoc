*COLD STORAGE* — Jamais chargé automatiquement. Consultation humaine uniquement.
Documentation exhaustive de la stratégie de gouvernance agent Eager/Lazy, Hot/Warm/Cold et du split encyclopédique.
Version originelle : `AGENT_MODUS_OPERANDI.adoc` (627 lignes) — splitée en `LAZY_EAGER_ESSENTIALS.adoc` (EAGER, 74 lignes) + ce fichier (COLD, ~600 lignes).
Rien n'a été supprimé. Tout a été relocalisé.

== P1 — Vue d'Ensemble

=== Problème Résolu

Face à la *fuite de contexte* entre les sessions d'un agent IA (Opencode), cette méthodologie fournit un système de gouvernance persistant basé sur des fichiers AsciiDoc, avec une dichotomie Eager/Lazy pour optimiser la consommation du token de contexte.

Le problème fondamental : l'agent ne se souvient pas de la session précédente. L'architecture, les conventions, l'état du backlog — tout est perdu. Reconstruire ce contexte coûte 30-40% du temps de session.

=== Principes Fondamentaux

. *EAGER (Tableau de Bord)* — Tout ce qui est critique et fréquemment utilisé doit être immédiatement accessible. Chargé automatiquement en début de session.
. *LAZY (Manuel du Propriétaire)* — Tout ce qui est contextuel ou volumineux doit être chargé sur demande.
. *HOT/WARM/COLD* — Mécanisme de vieillissement qui empêche le contexte de grossir indéfiniment sans rien perdre.

=== Analogie

L'approche s'inspire de la gestion de cache informatique :
- *EAGER* = cache L1 (immédiat, critique, petite taille)
- *LAZY* = cache L2 (sur demande, plus volumineux)
- *COLD (backup/)* = stockage froid (snapshots horodatés)
- *COLD (encyclopedies/)* = bibliothèque de référence (jamais auto)

=== Politique de Chargement

. *Début session* : `AGENT.adoc` (auto via opencode.json) + `PROMPT_REPRISE.adoc` + `LAZY_EAGER_ESSENTIALS.adoc` + `.agents/INDEX.adoc` + `.agents/SESSIONS_HISTORY.adoc` → ~800 lignes EAGER
. *Besoin détails* : charger `PROCEDURES.adoc`, `SESSION_CHECKLIST.adoc`, `AGENT_SESSION_MANAGER.adoc`, `sessions/` en LAZY
. *Connaissance froide* : `encyclopedies/`, `backup/` — jamais automatique, consultation humaine

== P3 — Règles Absolues

*Git* — Interdictions formelles : `git add`, `git commit`, `git push`, `git commit --amend`

*Fichiers de configuration* — Règle née de l'incident `site.yml` (bakery-plugin, session 2) :
. Vérifier `git check-ignore` + `git ls-files` AVANT toute modification
. Backup pré-modification OBLIGATOIRE
. Modifier avec `Edit` (partiel), PAS `Write` (écrasant)
. JAMAIS remplacer valeurs réelles par placeholders

*Tests* — Interdiction de lancer en procédure de fin de session sans permission explicite

*Build* — `publishToMavenLocal` obligatoire après chaque modification du code source (plugin Gradle)

*Secrets et tokens* — JAMAIS affichés dans le terminal (streaming possible sur Discord/Twitch)

*Modèle IA* — Calibré pour DeepSeek-V4-Pro (CSA+HCA, stable 128K). Ne pas remplacer par Kimi K2.6 (MLA pur, dégrade à 60K).

== P4 — Cycle de Vie d'une Session

=== Ouverture

*Startup prompt* (envoyé par l'utilisateur) :
```
Reprends le contexte du projet via les fichiers de gouvernance (AGENT.adoc, PROMPT_REPRISE.adoc, LAZY_EAGER_ESSENTIALS.adoc, INDEX.adoc) et propose le point de départ.
```

*Vérifications* :
```bash
./gradlew :newpipe:build
./gradlew sessionStatus
```

=== Pendant la Session

*Règles de travail* :
- 1 fichier à la fois (max 3 par session)
- `publishToMavenLocal` après chaque modif de .kt
- Proposer split si > 3 fichiers ou > 50 messages
- Consulter le backlog dans `AGENT.adoc` ou `.agents/MIGRATION_REFERENCE.adoc` (US-1 à US-9)

=== Fin de Session

*Trigger* : « fin de session », « on arrête là », « je quitte »

*Exécution silencieuse des 6 étapes* :

. *Archive* : créer `.agents/sessions/{N}-{type}-{sujet-kebab-case}.adoc` avec Contexte, Actions, Résultats, Build & Tests, Prochaine Session
. *PROMPT_REPRISE* : mettre à jour pour N+1 (résumé N, mission N+1, max 70 lignes)
. *SESSIONS_HISTORY* : ajouter ligne N / Date / Type / Sujet / Score X/10
. *INDEX* : mettre à jour sessions récentes, roadmap, portfolio
. *TEST_COVERAGE* : mettre à jour si tests ajoutés ou modifiés
. *COMPLETED_TASKS* : archiver tâches terminées

*Étape 7 optionnelle* : rotation backup si N%10==0 OU EAGER > 500 lignes

*Checklist finale obligatoire* — l'agent affiche la confirmation :
```
✅ Procédure de fin de session exécutée
📋 Checklist :
[✅] 1. Archive session N créée
[✅] 2. PROMPT_REPRISE.adoc mis à jour pour session N+1
[✅] 3. SESSIONS_HISTORY.adoc mis à jour
[✅] 4. INDEX.adoc mis à jour
[✅] 5. TEST_COVERAGE_ANALYSIS.adoc mis à jour (si applicable)
[✅] 6. COMPLETED_TASKS_ARCHIVE.adoc mis à jour
[✅] 7. Backup roté (si applicable)
```

== P5 — Métriques et Seuils

=== Session Idéale

|===
| Métrique | Cible
| Durée | 15-30 minutes
| Fichiers modifiés | 1-3 maximum
| Échanges LLM | 5-10 messages
| Contexte tokens | < 50k
===

=== Signes d'Alerte

. Agent répète des erreurs déjà corrigées → contexte saturé
. Plus de 3 fichiers modifiés → scope trop large
. Plus de 50 messages → attention diluée
. Plus de 3 échanges sans progrès → enlisement
. Contexte > 70k tokens → zone de dégradation
. Agent propose modifier newpipe/ ET buildSrc/ simultanément → violation scope

=== Règle d'Or

*Mieux vaut 5 sessions de 20 minutes qu'une session de 2 heures avec debugging chaos.*

== P9 — Défense en Profondeur contre la Saturation

*3 Couches* :

. *EAGER/LAZY* — Structure par *disponibilité* (tableau de bord vs manuel du propriétaire)
. *HOT/WARM/COLD* — Structure par *fraîcheur* (10 actives vs références vs vagues froides)
. *ESSENTIALS/ENCYCLOPÉDIE* — Structure par *densité* (compact critique vs exhaustif froid)

*Métaphore des 3 Étagères* :
- *Étagère 1 (EAGER)* — le plan de travail. Ce dont j'ai besoin maintenant.
- *Étagère 2 (LAZY)* — la bibliothèque de consultation. Ce que je peux aller chercher.
- *Cave (backup/ + encyclopedies/)* — les archives froides. L'agent n'y entre jamais. L'humain y descend pour distiller.

== Annexe A — Glossaire

*EAGER* : Fichier chargé automatiquement en début de session. Tableau de bord.
*LAZY* : Fichier chargé sur demande explicite de l'agent. Manuel du propriétaire.
*HOT* : Sous-ensemble EAGER — les 10 sessions les plus récentes.
*WARM* : Sous-ensemble LAZY — références, procédures, agents spécialisés.
*COLD* : Stockage froid — backup/ (vagues) et encyclopedies/ (connaissance). Jamais auto.
*Sliding Window* : Fenêtre glissante de 10 sessions dans les fichiers EAGER.
*Vague Froide* : Copie intégrale d'un état passé de la gouvernance, sans transformation.
*Split Encyclopédique* : Séparation d'un document monolithique en ESSENTIALS (EAGER) + ENCYCLOPEDIE (COLD).
*Portefeuille* : Tableau transverse listant tous les projets avec la même méthodologie.
*Capteur* : Mécanisme à 3 déclencheurs pour la rotation backup (auto/seuil/manuel).
*Rotation Backup* : Procédure qui migre les sessions N-10 et antérieures vers backup/.

        this.group = group

== Règles Absolues

=== 0. Build & Publication

*publishToMavenLocal obligatoire* après chaque modification du code source du plugin.
Toute modification dans `newpipe/newpipe/src/main/kotlin/` → republier le JAR local avant de tester.

```bash
./gradlew :newpipe:publishToMavenLocal
```

=== 1. Git — Interdictions Formelles

❌ `git add`, `git commit`, `git push`, `git commit --amend`
✅ Lecture seule autorisée : `git status`, `git diff`, `git log`

Seul l'utilisateur gère Git après validation humaine.

=== 1b. Fichiers de Configuration — Règle Absolue de Sûreté

*NÉE DE L'INCIDENT site.yml (bakery-plugin, session 2)* :

❌ *NE JAMAIS* écraser un fichier de config avec un `Write` complet quand un `Edit` partiel suffit
❌ *NE JAMAIS* remplacer des valeurs sensibles (tokens, secrets) par des valeurs factices ou placeholders
✅ Vérifier `git check-ignore` et `git ls-files` *avant* toute modification
✅ Utiliser le motif de backup pré-modification copié de `.agents/PROCEDURES.adoc`

Fichiers critiques :
- `local.properties` (secrets OAuth)
- `gradle.properties` (configuration build)
- `sessions.yml`, `musics.yml` (données utilisateur)

=== 2. Tests en Fin de Session — Interdiction Formelle

❌ *NE JAMAIS* lancer de tests en procédure de fin de session sans permission explicite.
La procédure de fin de session archive le contexte, elle ne valide pas le code.

=== 3. Sécurité des Tokens — Interdiction de Log

*PRINCIPE* : Les tokens API, mots de passe, refresh tokens ne doivent *JAMAIS* apparaître dans la sortie terminal.

❌ *NE JAMAIS* passer un token en argument visible d'une commande shell
❌ *NE JAMAIS* afficher le contenu de `local.properties`, `sessions.yml` ou tout fichier contenant des secrets
✅ Utiliser stdin pour l'injection de tokens

=== 4. Dépendance au Modèle IA

Le contexte EAGER (~800 lignes) est calibré pour *DeepSeek-V4-Pro* (stable jusqu'à 128K tokens, CSA+HCA).
Ne *PAS* remplacer par un modèle MLA pur (Kimi K2.6) qui dégrade au-delà de 60K tokens — voir article 0112.

DeepSeek-V4-Pro a été sélectionné après 13 sessions de test comparatif face à Kimi K2.6 et GLM-5.1.

=== Hot/Warm/Cold

*Hot (EAGER)* — 10 dernières sessions, PROMPT_REPRISE, règles, ~800 lignes
*Warm (LAZY)* — Références, procédures, agents spécialisés
*Cold (backup/)* — Vagues froides identiques (copie intégrale, jamais transformée)
*Cold (encyclopedies/)* — Connaissance thématique exhaustive, humain seulement

=== Rotation Backup

Déclencheurs :
. *Auto* : N % 10 == 0 (toutes les 10 sessions)
. *Seuil* : EAGER cumulé > 500 lignes
. *Manuel* : mot-clé « rotation backup »

La rotation crée `backup/Y20XX-sessions-X-Y/` avec copie intégrale de INDEX + HISTORY + COMPLETED_TASKS + sessions/, puis tronque les actifs à 10 sessions.

== Authentification Google — Contexte

*Setup* :
. Créer projet Google Cloud Console
. Activer YouTube Data API v3
. Créer credentials OAuth2 « Desktop app »
. Placer `client_secrets.json` dans `client_secrets/`

*Flux* :
```
buildSessions → authSessions (Device Flow) → sessions.yml enrichi → downloadMusic
```

*Sécurité* :
- `sessions.yml` et `client_secrets/` dans `.gitignore`
- `accessToken` en mémoire uniquement, jamais persisté
- `refreshToken` valable 1 an

== Procédure de Fin de Session

Mots-clés déclencheurs : « fin de session », « on arrête là », « je quitte », « fin de la session ».

L'agent exécute *silencieusement* les 6 étapes, puis affiche la checklist de confirmation.

. Créer `.agents/sessions/{N}-{type}-{sujet-kebab-case}.adoc`
. Mettre à jour `PROMPT_REPRISE.adoc` pour session N+1
. Mettre à jour `.agents/SESSIONS_HISTORY.adoc`
. Mettre à jour `.agents/INDEX.adoc`
. Mettre à jour `.agents/TEST_COVERAGE_ANALYSIS.adoc` (si tests modifiés)
. Mettre à jour `.agents/archives/COMPLETED_TASKS_ARCHIVE_YYYY-MM.adoc`
. Optionnel : rotation backup si N%10==0 OU EAGER cumulé > 500 lignes

Checklist de confirmation :
```
✅ Procédure de fin de session exécutée
📋 Checklist :
[✅] 1. Archive session N créée
[✅] 2. PROMPT_REPRISE.adoc mis à jour pour session N+1
[✅] 3. SESSIONS_HISTORY.adoc mis à jour
[✅] 4. INDEX.adoc mis à jour
[✅] 5. TEST_COVERAGE_ANALYSIS.adoc mis à jour (si applicable)
[✅] 6. COMPLETED_TASKS_ARCHIVE.adoc mis à jour
[✅] 7. Backup roté (si applicable)
```

*Règle absolue* : Aucune étape marquée [✅] si le fichier n'a pas été effectivement modifié et vérifié.
